# Mailcow SSL Sync and SNI Configuration Script

This script automates the synchronization of independent Let's Encrypt SSL certificates generated by an external ACME Companion to Mailcow container volumes. It also creates the corresponding SNI configurations (`domains` metadata files) so that Dovecot (IMAP/POP3) and Postfix (SMTP) can support independent certificates across multiple domains.

## Background

For security reasons, Mailcow's Web UI (WebUI) is not exposed to the public network (port 80 and 443 access from the public network is restricted or not mapped).

Since Mailcow's built-in ACME container only supports the HTTP-01 challenge, it cannot automatically request and renew certificates without exposing port 80. Therefore, the setting in `mailcow.conf` is configured to `SKIP_LETS_ENCRYPT=y`.

To address the need for automatic certificate renewal, this setup utilizes external [nginx-proxy](https://github.com/nginx-proxy/nginx-proxy) and [acme-companion](https://github.com/nginx-proxy/acme-companion) containers to automatically request and renew certificates via the DNS-01 challenge (e.g., Cloudflare DNS API). This script serves as an alternative solution to securely synchronize the externally renewed certificates to Mailcow and apply the corresponding SNI configurations under this architecture.

## Features

* **Independent Certificate Sync**: Automatically syncs `fullchain.pem` and `key.pem` from the Nginx Proxy certificate directory to the Mailcow SSL directory.
* **Secure Permission Configuration**: Fixes certificate file permissions after sync (certificates set to `644`, private keys set to `600`).
* **Automated SNI Mapping**: Automatically generates `domains` metadata files under each domain subdirectory, triggering Mailcow to generate `sni.conf` (Dovecot) and `sni.map` (Postfix) upon startup.
* **Minimized Downtime**: Restarts `mailcow-postfix`, `mailcow-dovecot`, and `mailcow-nginx` only when certificate or domain configuration changes are detected.

## Automated SNI Mapping?

If multiple mail domains (e.g., `mail.domain1.com`, `mail.domain2.com`, `mail.domain3.com`) share a mail server, Mailcow would use a single wildcard certificate or a Multi-Domain (SAN) certificate. To maintain privacy and isolation between different domains, this solution requests independent Let's Encrypt certificates for each mail domain. Under this multi-certificate architecture:

1. **Importance of SNI**: Since different domains use different certificates and private keys, Mailcow's Dovecot (IMAP/POP3) and Postfix (SMTP) must dynamically serve the correct SSL certificate based on the server name (SNI) sent by the client during connection.
2. **Mailcow's Automated Mapping Mechanism**: When Mailcow containers start up, their built-in initialization scripts scan all subdirectories under `/etc/ssl/mail/` (which maps to `assets/ssl/` on the host). If a subdirectory contains a `domains` metadata file, Mailcow automatically reads its content and generates the corresponding SNI mappings (`sni.conf` for Dovecot and `sni.map` for Postfix).
3. **The Script's Role**: This script automatically generates the correct `domains` metadata file in each certificate subdirectory during synchronization, thereby triggering and completing Mailcow's automated SNI mapping configuration.

## Requirements

### System Environment

* Linux operating system
* Docker and Docker Compose environment
* Deployed Mailcow mail server

### Certificate Source

* Independent SSL certificates obtained via DNS-01 challenge (e.g., Cloudflare API) using Nginx Proxy ACME Companion.
* Certificates stored in the Nginx Proxy container volume (default path: `/var/lib/docker/volumes/nginx-proxy-certs/_data/`).

### Deployment Notes

The following configurations and paths require special attention and adjustment before usage:

1. **Host Docker Volume Paths**:
   * The default paths `/var/lib/docker/volumes/...` assume that the host uses the default Docker Root Directory and named Docker volumes.
   * If a custom Docker data root directory is configured (e.g., `/home/docker/` or mounted to a separate drive), or if bind mounts are used (e.g., directly mounting `./data/assets/ssl` relative to the Mailcow path), ensure that the source and target paths in the script are updated to the actual absolute paths.
2. **Mailcow Container Names**:
   * The restart command in the script is `docker restart mailcow-postfix mailcow-dovecot mailcow-nginx`.
   * If the project name (`COMPOSE_PROJECT_NAME`) is customized in `mailcow.conf`, or if the stack is deployed via Portainer with a custom stack name, the container names might be prefixed differently (e.g., `mailcow-postfix-1`, `my-mail-postfix`). Check the actual container names using `docker ps` and update the `docker restart` command in the script accordingly.

## Installation

1. Copy the script to the system execution path:

   ```bash
   sudo cp main.sh /usr/local/bin/sync-mailcow-ssl.sh
   sudo chmod +x /usr/local/bin/sync-mailcow-ssl.sh

   # Replace /usr/local/bin with the desired path for storing the script
   ```

## Script Customization

Before usage, the synchronization items in `main.sh` must be modified according to the actual environment. The sections that require modifications are the `sync_cert` calls at the bottom of the script:

```bash
# Format:
# sync_cert "external_source_cert_directory" "mailcow_target_cert_directory" "corresponding_domain_name"

# 1. Sync the default mail certificate (fallback certificate used when client does not send SNI)
sync_cert \
  "/var/lib/docker/volumes/nginx-proxy-certs/_data/mail.domain1.com" \
  "/var/lib/docker/volumes/mailcow_data/_data/assets/ssl" \
  "mail.domain1.com"

# 2. Sync the SNI certificate for Domain 1
sync_cert \
  "/var/lib/docker/volumes/nginx-proxy-certs/_data/mail.domain1.com" \
  "/var/lib/docker/volumes/mailcow_data/_data/assets/ssl/mail.domain1.com" \
  "mail.domain1.com"

# 2. Sync the SNI certificate for Domain 2
sync_cert \
  "/var/lib/docker/volumes/nginx-proxy-certs/_data/mail.domain2.com" \
  "/var/lib/docker/volumes/mailcow_data/_data/assets/ssl/mail.domain2.com" \
  "mail.domain2.com"
```

The following parts need to be modified:

1. The external certificate source path (e.g., replace `/var/lib/docker/volumes/nginx-proxy-certs/_data/` with the actual certificate storage path).
2. The Mailcow container volume path (e.g., replace `/var/lib/docker/volumes/mailcow_data/_data/` with the actual Mailcow data directory).
3. The domain name string parameter passed to each call (e.g., `mail.domain1.com`).

## Execution and Automation

### Manual Execution

```bash
sudo /usr/local/bin/sync-mailcow-ssl.sh
```

### Scheduled Automation

Automated sync via cron is recommended to detect and apply renewed certificates.

1. Edit the system crontab:

   ```bash
   sudo crontab -e
   ```

2. Add the following entry (executed daily at 3:00 AM):

   ```cron
   0 3 * * * /usr/local/bin/sync-mailcow-ssl.sh >/dev/null 2>&1
   ```

## Directory Structure Reference

After synchronization, the Mailcow SSL assets directory is structured as follows:

```text
/var/lib/docker/volumes/mailcow_data/_data/assets/ssl/
├── cert.pem (Default certificate)
├── key.pem (Default private key)
├── domains (Default certificate domain metadata)
├── mail.domain1.com/ (Domain 1)
│   ├── cert.pem
│   ├── key.pem
│   └── domains
├── mail.domain2.com/ (Domain 2)
│   ├── cert.pem
│   ├── key.pem
│   └── domains
└── mail.domain3.com/ (Domain 3)
    ├── cert.pem
    ├── key.pem
    └── domains
```
